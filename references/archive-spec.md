# 小说存档与连续性追踪系统

## 设计哲学

存档系统由**两层**组成，缺一不可：

| 层级 | 思维模式 | 记录什么 | 解决什么问题 |
|------|----------|----------|-------------|
| **数据库层** | 数据思维 | 事实、数值、状态 | 数据前后矛盾、数值不一致 |
| **读者层** | 读者思维 | 因果、理解、疑问 | 人物动机不合理、剧情逻辑崩塌 |

**数据库记录"是什么"，读者层记录"为什么"。**

审核新章节时，两层交叉验证：
- 数据库说灵石余额250，新章节说只剩10块 → **数据不一致**
- 读者层记录"张三性格谨慎"，新章节写他冲动赌博 → **人物崩了**

## 数据库层（保留原有格式）

以下文件保持不变：characters.md、hooks.md、foreshadowing.md、timeline.md、settings.md

这些文件用**表格和编号**追踪事实，专注于：
- 数值变动（灵石、等级、属性）
- 状态变更（存活/死亡/失踪）
- 时间线一致性
- 伏笔/钩子的状态

### 为什么需要这一层？

数据库能告诉你"张三获得100灵石"，但不能告诉你"为什么"。

读者笔记记录的是：
- **动机**：张三获得灵石是因为系统发放，还是因为他完成了任务？
- **性格**：张三拿到灵石后的反应是什么？高兴？无感？这说明他是什么样的人？
- **关系**：这100灵石改变了谁和谁的关系？
- **直觉**：这段读起来怪怪的，但我说不上来哪里不对

这些信息是**审核逻辑问题和人物一致性问题的关键依据**。

## 两层交叉验证流程

审核新章节时，执行以下交叉检查：

### 1. 数据库 → 数据库（原有）
```
检查数值是否前后一致
检查人物状态是否矛盾
检查时间线是否连贯
```

### 2. 数据库 → 读者层（新增）
```
数据库记录的事实，读者层有没有理解？
  例：数据库记录"张三获得100灵石"，但读者层没提
  → 可能是读者忽略了，也可能是剧情不重要

读者层的理解，数据库能不能支撑？
  例：读者层写"张三很谨慎"，但数据库没有相关行为记录
  → 需要确认这个"谨慎"的印象来自哪里
```

### 3. 读者层 → 读者层（新增）
```
新章节的理解和之前的理解是否矛盾？
  例：第3章读者层写"张三信任师父"
  第15章读者层写"张三开始怀疑师父"
  → 这是正常的角色成长，但需要确认是否有铺垫

新章节是否回答了之前的疑问？
  例：第1章读者层写"骸骨说海里怎么会有这种东西？"
  第5章揭晓答案 → 更新疑问状态
```

### 4. 读者层 → 新章节（审核核心）
```
新章节中角色的行为，和读者层记录的性格/动机是否一致？
  例：读者层记录"张三性格谨慎，不会冒险"
  新章节：张三主动挑战比自己强的敌人
  → 如果没有充分的动机铺垫，这是人物崩塌
```

## 存档更新流程（完整版）

```
新章节输入
    ↓
Step 1: 数据库录入
  - 更新 characters.md（人物状态）
  - 更新 timeline.md（数值变动）
  - 更新 hooks.md / foreshadowing.md
  - 更新 settings.md（如有新设定）
  - 更新 index.md

Step 2: 读者理解录入
  - 写 chapters/chXXX.md 的"读者理解"部分
  - 更新 reader-notes.md（追加本章的理解）
  - 如果有新的因果/动机/关系变化，写入对应章节

Step 3: 交叉验证
  - 数据库 vs 数据库
  - 数据库 vs 读者层
  - 读者层 vs 读者层
  - 输出连续性检查报告

Step 4: 追加 log.md
  - 记录本次操作
```
