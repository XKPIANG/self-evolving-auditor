# Novel Audit — 自进化小说审核系统

## 核心理念

这个skill不是静态工具，而是一个**会学习的审核员**：
- 拆解一本书 → 提取知识点 → 评估价值 → 写入知识库
- 知识库有版本号，每次学习都会升级
- 低价值知识会被遗忘，重复知识会被整合
- 拆解越多，审核越准

## 核心理念：编译式审核（借鉴 Karpathy LLM Wiki）

传统审核方式（RAG式）：每次审核都要重新读小说、重新找问题、重新分析——没有积累。

本系统采用**编译式**方式：每读一章，就把它「编译」进一个活的 Wiki。后续审核不需要重新发现，直接查 Wiki 就知道所有人物状态、设定规则、伏笔钩子。

**核心区别：**
| | RAG式（传统） | 编译式（本系统） |
|--|--|--|
| 每次审核 | 重读全文找问题 | 查 Wiki 对比新章节 |
| 知识积累 | 无，每次从零 | 持续积累，越用越准 |
| 连续性检查 | 靠记忆或手动翻 | Wiki 自动追踪 |
| 跨章节关联 | 容易遗漏 | 交叉引用自动维护 |

**编译流程：**
```
新章节 → 读取 → 提取信息 → 更新 Wiki 页面 → 输出审核结果
                    ↓
          人物页面更新 / 设定页面更新 / 伏笔页面更新 / 数据变动记录
                    ↓
              index.md 索引更新 / log.md 操作日志
```

**Wiki 不是给作者看的文档，是给审核 AI 用的工作台。**

## 工作流

### 流程一：拆解样板书（学习模式）

输入：txt 文件路径 + 书名

**Step 1：通读分析**
- 章节划分 & 字数统计
- 节奏曲线绘制（每章情绪值 -5 到 +5）
- 爽点标注（位置、类型、铺垫、释放）
- 逻辑框架提取（世界观、能力体系、角色关系）
- 文风特征提取

**Step 2：提取知识点**
- 从分析结果中提炼可复用的模式
- 每个知识点包含：模式描述、来源书目、置信度、标签

**Step 3：知识评估与入库**

对每个提取的知识点：

```
1. 查重：knowledge.json 中是否已有相似条目？
   ├── 有 → 合并（times_seen++，confidence 更新，补充新示例）
   └── 无 → 评估价值后决定是否入库

2. 价值评估（0-1 分）：
   - 可复用性：这个模式能用在其他书上吗？
   - 具体性：描述够具体吗？（"写得好看"=0分，"3章一爽点"=0.8分）
   - 反面价值：知道"什么是坏的"也有价值

3. 入库决策：
   confidence >= 0.6 → 入库
   confidence < 0.6 → 丢弃（不值得占知识库空间）
```

**Step 4：知识整合（去重 & 升级）**

每次入库后运行整合：
```
遍历所有知识条目：
  1. 找相似条目（相同类型+相似描述）
  2. 合并：
     - times_seen = sum
     - confidence = weighted_average（按 times_seen 加权）
     - tags = union
     - 示例 = 保留最好的
  3. 标记待遗忘的条目（见下方遗忘机制）
```

**Step 5：遗忘机制**

```
每分析 N 本书后（N = 已分析总数的 20%，至少 3 本），运行遗忘检查：

遍历所有条目：
  if times_seen == 1 且 距上次更新 > 5本书：
    → 标记为"待遗忘"
    → 如果连续2次检查都是待遗忘 → 删除
  
  if confidence < 0.3：
    → 直接删除（低价值知识不值得保留）
```

**Step 6：输出拆解报告 + 学习报告**

```
# 样板书拆解报告
[完整分析内容]

# 学习报告
- 新增知识点：X 个
- 整合已有知识：X 个
- 遗忘低价值知识：X 个
- 知识库版本：vX → vY
- 知识库总条目：X
```

### 流程二：小说审核（应用模式）

输入：小说文本 + 目标平台（番茄/起点，默认番茄）

**Step 0：平台识别**
- 用户指定平台 → 使用对应标准
- 用户未指定 → 询问目标平台，或根据文风自动判断
- 番茄特征：快节奏、短章节、打脸爽文为主
- 起点特征：较长章节、世界观宏大、文笔要求高

**Step 1：加载知识库**
- 读取 `knowledge/knowledge.json`
- 读取 `references/` 基础框架
- 加载对应平台标准（`standards-fanqie.md` 或 `standards-qidian.md`）
- 合并为当前审核标准

**Step 2：六维审核**

用知识库中的模式 + 基础框架 + 平台标准进行审核：

- **节奏**：用 rhythm_patterns 检查节奏是否符合已学模式（参照平台节奏标准）
- **爽点**：用 hook_patterns 检查爽点设计是否到位（参照平台爽点偏好）
- **逻辑**：用 logic_rules + anti_patterns 检查硬伤
- **发布水平**：用对应平台 standards + platform_insights 检查
- **AI味检测**：用 ai-style.md 检测 AI 生成痕迹，标注命中项和修改建议

**Step 3：输出审核报告**

格式同 v1，但增加：
- 标注使用了哪些知识条目（让用户知道审核依据）
- 标注适用的平台标准
- 如果发现了知识库中没有的新模式，提示用户是否学习

### 流程五：快速审核模式（优化版）

**目标**：一次工具调用完成审核+存档更新，减少重复读取。

**与流程二的区别**：
- ❌ 不重复读 references/ 文件（知识库已包含所有模式）
- ❌ 不逐个写存档文件（用一次 exec 批量写入）
- ❌ 不写冗长分析（直接出结论+关键引用）
- ✅ 只读 knowledge.json + 现有存档（characters/settings/timeline/hooks/foreshadowing）
- ✅ 审核报告精简为：六维评分 + 3-5条关键批注 + 连续性检查 + 建议
- ✅ 存档更新用 Python 脚本一次性写完

**快速审核流程**：

```
Step 1: 一次性读取
  - knowledge.json（知识库模式）
  - archives/{书名}/characters.md + settings.md + timeline.md + hooks.md + foreshadowing.md（现有存档）
  - 如果是新书，创建存档目录

Step 2: 审核（不调用工具，直接分析）
  - 六维评分（节奏/爽点/逻辑/发布水平/设定一致性/AI味检测）
  - 连续性检查（对比现有存档数据）
  - 关键批注（3-5条，每条一句话）

Step 3: 批量更新存档
  - 用一次 exec 调用 Python 脚本，一次性写入所有需要更新的文件
  - 输出：新/更新的文件列表 + 审核报告

Step 4: 推送（可选）
  - git add + commit + push
```

### 流程四：存档模式（编译式 Wiki 追踪）

输入：小说章节文本 + 章节号 + 书名

**核心思想：** 不是每次审核都重读全文，而是每读一章就「编译」进 Wiki，后续审核直接查 Wiki 对比新章节。

**四大操作：**

#### 操作一：录入（Ingest）—— 新章节编译进 Wiki

审核新章节时，同步执行：

```
1. 创建 chapters/chXXX.md（章节档案）
2. 更新 characters.md（人物状态变化）
3. 更新 timeline.md（财产/等级/物品变动）
4. 更新 settings.md（新确立的规则/设定）
5. 更新 hooks.md（新钩子 / 钩子推进 / 钩子解决）
6. 更新 foreshadowing.md（新伏笔 / 伏笔回收）
7. 更新 index.md（索引更新）
8. 追加 log.md（操作日志）
```

一章可能触发 5-10 个 Wiki 页面的联动更新。这是「编译」的核心——一次录入，全局联动。

#### 操作二：查询（Query）—— 审核时查 Wiki

审核新章节时，不是重读全文，而是：

```
1. 读 index.md → 找到相关页面
2. 读 characters.md → 当前人物状态
3. 读 timeline.md → 最新数据
4. 读 settings.md → 已确立的规则
5. 读 hooks.md / foreshadowing.md → 未解决的悬念
6. 对比新章节内容 → 输出审核结果
```

好处：审核第50章时，不需要重读前49章，Wiki 已经编译好了所有关键信息。

#### 操作三：回填（File Back）—— 审核结果写回 Wiki

审核产生的有价值信息，应该写回 Wiki：

```
- 审核发现的新模式 → 写入 topics/ 或 knowledge/
- 对比分析结果 → 写入 topics/compare-xxx.md
- 人物关系图谱 → 写入 topics/relationships.md
- 剧情预测 → 写入 topics/predictions.md
```

你的每一次审核，都在让 Wiki 变得更丰富。用得越多，它越聪明。这不是消耗，是投资。

#### 操作四：自检（Lint）—— 定期体检

定期（每 5-10 章或用户主动触发）让系统做一次全面体检：

```
1. 数据一致性：所有数值是否前后连贯？
2. 设定一致性：有没有规则被违反？
3. 人物一致性：性格/立场是否前后矛盾？
4. 伏笔健康度：是否有伏笔超过 20 章未回收？
5. 钩子健康度：是否有钩子被遗忘？
6. 孤立页面：是否有 Wiki 页面没有任何引用？
7. 信息空缺：是否有可以补全但未补全的信息？
8. 交叉引用：相关页面之间是否建立了链接？
```

输出健康报告，标记所有问题，给出修复建议。
